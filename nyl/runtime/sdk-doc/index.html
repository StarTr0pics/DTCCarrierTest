    SDK Description

    SDK allows to override/extend the following actions performed by SD Runtime:

    • saveData - invoked when Runtime wants to save client data on server;
    • loadData - invoked when Runtime wants to load client data from server asynchronously;
    • callCustomAssembly - invoked when Runtime calls a 3rd party service to perform calculations or retrieve some data;
    • callJsCustomAssembly - invoked when Runtime calls a custom javascript service to execute any custom logic;
    • getSourceGridData - invoked when Runtime requests data for a specific Source Grid control;
    • getSourceDropdownData - invoked when Runtime requests data for a specific Source Dropdown control;
    • validateCreditCard - invoked to validate credit card data via specific services;
    • changeCreditCard - invoked to erase previously entered credit card data;
    • verifyEmail - invoked whenever email control makes a call to validate email;
    • viewPDF - invoked whenever event action "View PDF" happened. This handler executes RuntimeCoreAPI.showModal method and pass there a component with pdf iframe inside. You could override this handler and show your own modal window components;
    • uploadFile - invoked whenever Runtime attempts to upload file;
    • showAttachments - invoked whenever Runtime attempts to show attachments;
    • onDynamicTableEvent - invoked as a reaction on events happening inside dynamic table control;

    Names of available actions are defined in the API constant global.App.runtimeCoreApi.sdkEventHandlerNames. Detailed documentation of SDK actions is available in GlobalEventHandlers section of this guide.

    Usage

    SDK provides a JavaScript API to override/extend specified actions, access Runtime data, call 3rd party services, transform data, etc.

    SDK API is available via global JS function createGlobalEventHandlersExtension(ExtensionClass). The function is called by runtime upon its initialization and supplies the ExtensionClass parameter, which can be used to access/override implementation of SDK actions.

    The createGlobalEventHandlersExtension function must be declared before the runtime-xxx.js script.

    Example: The "scripts.html" file inside "application" folder of the downloaded runtime could be modified in a way:

    <!-- Load custom script first -->
<script>
    function createGlobalEventHandlersExtension(ExtensionClass) {
        // your code here
    }
</script>

<!-- External libraries  -->
<script src="{{staticsPath}}/assets/jquery-3.3.1.min.js" defer></script>
<script src="{{staticsPath}}/assets/bootstrap-3.3.7.min.js" defer></script>

<!-- Then load runtime.js -->
<script src="{{staticsPath}}/assets/runtime.js" defer></script>

    External libraries

    If there is necessity in external libraries load them in scripts.html. By default it contains jquery-3.3.1.min.js and bootstrap-3.3.7.min.js. Replace these links if you need another versions of libraries, but note that runtime.js is tested with 2.1.3 and 3.3.1 versions of jQuery and 3.3.7 of bootstrap. Using other versions is possible but stable work is not guaranteed. Remember that runtime.js doesn't work without jQuery and bootstrap.

    In global scope available next libraries:

    • React v15.1.0 https://github.com/facebook/react/tree/v15.0.1/docs
    • ReactBootstrap v0.28.5 https://github.com/react-bootstrap/react-bootstrap/tree/v0.28.5/docs
    <!-- Examples -->
<script>
    function createGlobalEventHandlersExtension(ExtensionClass) {
        React.createElement(ReactBootstrap.Modal,
        {
            autoFocus: true,
            animation: false,
            show: true,
            backdrop: 'static',
            onHide: hideModal,
            bsSize: 'large'
        })
    }
</script>

    Default implementation

    In order to keep default implementation of any action just don't override corresponding SDK action or even don't implement the createGlovalEventHandlersExtension function.
    By default all methods inside ExtensionClass don't update UI (re-render view according to updated state). They are called inside EventManager which manages asynchronous function calls and updates UI automatically after completing calling current stack of asynchronous functions. In order to update UI whenever you need call refreshUI method of runtimeCoreApi.

    Extending default implementation

    Default implementation of any SDK action is available via ExtensionClass.superclass property. In order to extend default action behavior you need to redefine corresponding SDK action and use the ExtensionClass.superclass to access base implementation:

    function createGlobalEventHandlersExtension(ExtensionClass) {
    var callCustomAssemblyHandlerName =
        global.App.runtimeCoreApi.sdkEventHandlerNames.CALL_CUSTOM_ASSEMBLY;
    ExtensionClass.prototype[callCustomAssemblyHandlerName] = function(assemblyName) {
        // e.g. update PI_FirstName in runtime data
        this.coreApi.updateRuntimeData({
            caseData: [
                {
                    name: 'PI_FirstName',
                    value: 'Andrew',
                    formattedValue: 'Andrew',
                    isValid: 'true'
                }
            ],
            shouldReplaceSavedData: false
        });

        // call default implementation of callCustomAssembly and return the resulting promise,
        // which resolves in data provided by a custom assembly (if any)
        return this.superclass.callCustomAssembly(assemblyName);
    };
}

    Overriding default implementation

    In order to override default action behavior you just need to redefine a corresponding SDK action in ExtensionClass prototype:

    function createGlobalEventHandlersExtension(ExtensionClass) {
    var callCustomAssemblyHandlerName =
        global.App.runtimeCoreApi.sdkEventHandlerNames.CALL_CUSTOM_ASSEMBLY;
    ExtensionClass.prototype[callCustomAssemblyHandlerName] = function(assemblyName) {
        console.log("Don't really call:", assemblyName);
    };
}

    Runtime core API functions

    Runtime API functions are available through ExtensionClass.coreApi property (available only inside ExtensionClass methods). Also this API is exposed globally: global.App.runtimeCoreApi.
    Documentation of API methods is available in the API section of this guide.

    Embedding runtime into page

    To do that, on your web page wrapper element for runtime should be created. Plus, you have to design JS function which should find and return this wrapper element.

    Example: The next wrapper is created: <div id="runtime-entry"></div> To embed runtime application into this wrapper you need to create the next function:

    global.App.getRuntimeEntryPoint = function() {
    return document.querySelector('#runtime-entry');
};

    This function should be defined after global is defined.

    Working with JavaScript Custom Assemblies

    The JavaScript (JS) Custom Assemblies (CA) are services (technically just JS functions) which loaded directly to client as script and work without any network activity, unlike default custom assemblies. To load JS CA and make it available for the application it should be initialized in global.App.jsCA object by default:

    <script>
    (function () {
        function setFullNameCaFunction() {
            // this points to ExtensionClass
            const clientData = this.coreApi.getRuntimeData();

            const firstName = clientData.FirstName.value;
            const lastName = clientData.LastName.value;
            const fullName = firstName + " " + lastName;

            this.coreApi.updateRuntimeData({
                caseData: [
                    {
                        name: 'FullName',
                        value: fullName,
                        formattedValue: fullName
                    }
                ]
            });
        }

        window.global = window.global || {};

        global.App = global.App || {};

        global.App.jsCA = {
            setFullNameCa: setFullNameCaFunction
        }
    })();
</script>

    The format of global.App.jsCA object is the following:

    global.App.jsCA = { key: value };

    where: key - name of the JS CA which is set to call_js_service action in screen designer. value - JS function which should be executed when call_js_service action occurs.

    IMPORTANT JS CA function must be implement in synchronous way while using it inside Rule's call_js_service action. It possible to run asynchronous code ONLY for Event's call_js_service action.

    It is also possible to override default behaviour how to invoke JS CA. There is a method in SDK ExtensionClass called callJsCustomAssembly. Please check section Overriding default implementation to understand how to override it.

    Triggering Actions and Events

    General information

    Application uses messages to communicate with its container (for example if it's opened in an iframe). A container may send messages to the Application, thus requesting it to perform certain actions. In its turn the Application sends messages to container to notify it about events which happen inside the Application.

    Incoming Messages (handled by Application)

    In order to trigger an action in the Application, send a message to it. Syntax:

    applicationWindow.postMessage({ type: '<INCOMING_MESSAGE_TYPE>' [, data: <OPTIONAL_DATA_OBJECT>] }, '*');

    All incoming messages generate a "feedback" message, which provides a status of the requested action execution. See the "Outgoing Messages" section below with detailed description of these messages.

    Types of incoming messages:
    • BEFORE_LEAVE_SCREEN_REQUEST

    Triggers the "Before Leave" action on a current screen.

    applicationWindow.postMessage({ type: 'BEFORE_LEAVE_SCREEN_REQUEST' }, '*');
    • SAVE_CLIENT_DATA_REQUEST

    Triggers saving of client data.

    applicationWindow.postMessage({ type: 'SAVE_CLIENT_DATA_REQUEST' }, '*');
    • VIEW_PDF_REQUEST

    Triggers the "View PDF" action.

    applicationWindow.postMessage({ type: 'VIEW_PDF_REQUEST' }, '*');
    • LOAD_CLIENT_DATA_REQUEST

    Triggers loading of client data either from server or from provided parameter. In the latter case no server requests will be sent, provided values override current client data state.

    NOTE: You can override default implementation of data loading. See loadData method in GlobalEventHandlers.

    // load client data from server:
applicationWindow.postMessage({ type: 'LOAD_CLIENT_DATA_REQUEST' }, '*');

// load client data from provided data object:
applicationWindow.postMessage(
    {
        type: 'LOAD_CLIENT_DATA_REQUEST',
        clientData: {
            /* your data here */
        }
    },
    '*'
);
    • OPEN_SCREEN_REQUEST

    Switches to a screen with specified name.

    applicationWindow.postMessage(
    { type: 'OPEN_SCREEN_REQUEST', screenName: '<screen_name_to_open>' },
    '*'
);

    Outgoing Messages (sent by Application)

    In order to handle messages sent by the Application inside a container, use event listeners. Syntax:

    window.addEventListener('message', function(e) {
    if (e.data.type === OUTGOING_MESSAGE_TYPE) {
        // where OUTGOING_MESSAGE_TYPE is one of supported message types
        var messageData = e.data.data;
        // do something
    }
});
    Types of outgoing messages :
    • OPEN_SCREEN_REQUEST_SUCCESS, OPEN_SCREEN_REQUEST_FAILED

    "Feedback" messages for OPEN_SCREEN_REQUEST.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'OPEN_SCREEN_REQUEST_SUCCESS') {
        // handle successful case
    }
    if (e.data.type === 'OPEN_SCREEN_REQUEST_FAILED') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • BEFORE_LEAVE_SCREEN_SUCCESS, BEFORE_LEAVE_SCREEN_FAILED

    "Feedback" messages for BEFORE_LEAVE_SCREEN_REQUEST.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'BEFORE_LEAVE_SCREEN_SUCCESS') {
        // handle successful case
    }
    if (e.data.type === 'BEFORE_LEAVE_SCREEN_FAILED') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • SAVE_CLIENT_DATA_SUCCESS, SAVE_CLIENT_DATA_FAILED

    "Feedback" messages for SAVE_CLIENT_DATA_REQUEST.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'SAVE_CLIENT_DATA_SUCCESS') {
        // handle successful case
    }
    if (e.data.type === 'SAVE_CLIENT_DATA_FAILED') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • VIEW_PDF_SUCCESS, VIEW_PDF_FAILED

    "Feedback" messages for VIEW_PDF_REQUEST.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'VIEW_PDF_SUCCESS') {
        // handle successful case
    }
    if (e.data.type === 'VIEW_PDF_FAILED') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • LOAD_CLIENT_DATA_SUCCESS, LOAD_CLIENT_DATA_FAILED

    "Feedback" messages for LOAD_CLIENT_DATA_REQUEST.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'LOAD_CLIENT_DATA_SUCCESS') {
        // handle successful case
    }
    if (e.data.type === 'LOAD_CLIENT_DATA_FAILED') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • BEFORE_LOAD_SCREEN_FAILED, BEFORE_LOAD_SCREEN_SUCCESS

    "Feedback" messages for internal action "Before Load Screen", which cannot be triggered from an outside container and occurs whenever current screen switches to another one.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'BEFORE_LOAD_SCREEN_FAILED') {
        // handle successful case
    }
    if (e.data.type === 'BEFORE_LOAD_SCREEN_SUCCESS') {
        var error = e.data.data; // error object
        // handle error
    }
});
    • UPLOAD_FILE

    Fired when "Upload File" action is executed inside the Application.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'UPLOAD_FILE') {
        var defaultFileName = e.data.data; // default file name specified in the "UploadFile" action
        // process action
    }
});
    • SHOW_ATTACHMENT_LIST

    Fired when "Show Attachments" action is executed inside the Application.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'SHOW_ATTACHMENT_LIST') {
        // process action
    }
});
    • START_ESIG_UI

    Fired when "Start eSig UI" action is executed inside the Application.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'START_ESIG_UI') {
        // do appropriate stuff
    }
});
    • FINISH_ESIG_UI

    Fired when "Finish eSig UI" action is executed inside the Application.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'FINISH_ESIG_UI') {
        // do appropriate stuff
    }
});
    • MULTIPLE_EDITING_FAILED

    Fired when a server request returns the "MULTIPLE_EDITING_BLOCK" error.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'MULTIPLE_EDITING_FAILED') {
        // do appropriate stuff
    }
});
    • UPDATE_SESSION_TIMER_REQUEST

    Fired before each request to server to allow performing any required actions, e.g. update session timer.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'UPDATE_SESSION_TIMER_REQUEST') {
        // process event
    }
});
    • UPDATE_FIRST_AND_LAST_NAME_REQUEST

    Fired on any value change inside the application. Contains values of "PIFirstName" and "PILastName" client data items.

    window.addEventListener('message', function(e) {
    if (e.data.type === 'UPDATE_FIRST_AND_LAST_NAME_REQUEST') {
        var firstName = e.data.data.PIFirstName;
        var lastName = e.data.data.PILastName;
        // process retrieved values
    }
});
    • START_SCREEN_CHANGING

    Fired on navigation between screens. Indicates the start of the screen changing process. Contains name of the next screen as "screenName" and name of current screen which is going to become previous screen as "previousScreenName".

    window.addEventListener('message', function(e) {
    if (e.data.type === 'START_SCREEN_CHANGING') {
        var screenName = e.data.screenName;
        var previousScreenName = e.data.previousScreenName;
        // process event
    }
});
    • FINISH_SCREEN_CHANGING

    Fired on navigation between screens. Indicates the finish of the screen changing process. Contains name of the new screen "screenName".

    window.addEventListener('message', function(e) {
    if (e.data.type === 'FINISH_SCREEN_CHANGING') {
        var screenName = e.data.screenName;
        // process event
    }
});

    UI Configuration

    You can use configs/uiConfig.json file to adopt the user interface to your company's specific Requirements.

    Screen navigation panel

    The screen navigation panel's visibility is handled changing show_nav_panel property: true to show, false to hide. By default the screen navigation panel is visible(show_nav_pane: 'true').

    Backspace behavior

    Backspace to go back action in non-Chrome browsers could be controlled by disable_backspace_navigation property: true to disable moving backwards through the browser history, false to enable moving backwards through browser history. Note that Chrome disables going back by history by itself.

    Applying custom styles and fonts

    Exported application includes all required stylesheets by default. However you can replace any default stylesheet with custom one. To do it, open styles.html file and just update href attribute of particular <link> (HTML element) to point it to your custom resource. All default stylesheets are wrapped in comments with name of the resource that <link> element will load.

    Also note that all style link elements should be put inside the wrapping <noscript id="deferred-styles"> tag to allow loading them in a deferred manner to avoid blocking the runtime application HTML rendering. This is required for displaying the loading spinner.

    styles.html

    <noscript id="deferred-styles">
    <!--fonts-->
        <link href="{{staticsPath}}/assets/runtime-fonts-fb9249627a6a1adfb091.css" rel="stylesheet">
    <!--fonts-->

    <!--font-awesome-->
        <link href="{{staticsPath}}/assets/runtime-font-awesome-50cc4c94e470f2e05632.css" rel="stylesheet">
    <!--font-awesome-->

    <!--bootstrap-theme-->
        <link href="{{staticsPath}}/assets/runtime-bootstrap-theme-9a029d11b10303249111.css" rel="stylesheet">
    <!--bootstrap-theme-->

    <!--bootstrap-overrides-->
        <link href="{{staticsPath}}/assets/runtime-bootstrap-overrides-9a029d11b10304564654.css" rel="stylesheet">
    <!--bootstrap-overrides-->

    <!--bootstrap-controls-->
        <link href="{{staticsPath}}/assets/runtime-controls-518eaa3f2883aa62e72f.css" rel="stylesheet">
    <!--bootstrap-controls-->

    <!--runtime-styles-->
        <link href="{{staticsPath}}/assets/runtime-a501552b175aef3fa4f4.css" rel="stylesheet">
    <!--runtime-styles-->

    <!--browser-fixes-->
        <link href="{{staticsPath}}/assets/runtime-browser-fixes-fb9249627a6df67d7df22.css" rel="stylesheet">
    <!--browser-fixes-->
</noscript>

    If you want to update for example bootstrap theme just replace href attribute of <link> to use different URI. Same works for fonts and other styles.

    styles.html

    <!--bootstrap-theme-->
    <link href="https://maxcdn.bootstrapcdn.com/bootstrap/3.3.7/css/bootstrap.min.css" rel="stylesheet">
<!--bootstrap-theme-->

    CSS namespacing

    Runtime package contains css files which could conflict with parent web site ( in case of embedding runtime app into web page ). Therefore SDK provides ability to resolve such sort of conflicts via generating namespaced css files. In case you have css conflicts you need to manually change path to css files inside styles.html.

    Example:

    <link href="{{staticsPath}}/assets/runtime-d9062fc459f15ab12eab.css" rel="stylesheet">

    to

    <link href="{{staticsPath}}/assets/namespacedCss/runtime-d9062fc459f15ab12eab.css" rel="stylesheet">

    Be aware that runtime root html element (which is configured via global.App.getRuntimeEntryPoint see Embedding runtime into page section) should have specific css class. By default it's __ but maybe changed at screen designer build time.

    Dynamic table configuration

    This file describes configuration and behavior of dynamic table control.

    Table options

    • columns: Array<Column>
      • Required
      • The core columns configuration object for the entire table.
      • Supports nested columns arrays via the column's columns key, eg. [{ title: 'My Group', columns: [...] }]
    • data: Array<Row>
      • Required
      • The data array that you want to display on the table.
    • defaultColumn: Partial<Column>
      • Optional
      • Default options for all columns
      • options (see Column Options for description):
        • width
        • minWidth
        • maxWidth
        • cssClass
        • show

    • sortBy: Array<Object<name: column.name, desc: Bool>>

      • Optional
      • An array of sorting objects. If there is more than one object in the array, multi-sorting will be enabled. Each sorting object should contain an name key with the corresponding column name to sort by. An optional desc key may be set to true or false to indicated ascending or descending sorting for that column.

    • pagination: Object<page: Number, itemsPerPage: Number>

      • Optional
      • In case when this option is presented table control will draw pagination controls controlled by page and itemsPerPage passed in option object.

    • disabled: Boolean

      • Optional
      • Disable table functionality and all editors inside it

    Column Options

    The following options are supported on any column object you can pass to columns.

    • name: String
      • Required (but optional for grouping columns)
      • This string is used to build the data model for your column.
      • The data returned by an accessor should be primitive and sortable.
      • The column's value will be looked up on the original row via that key, eg. If your column's name is firstName then its value would be read from row[<rowIndex>]['data']['firstName'].
    • columns: Array<Column>
      • Optional
      • A nested array of columns.
      • If defined, the column will act as a header group. Columns can be recursively nested as much as needed.
    • title: String
      • Optional
      • The name to render in the column header
    • show: Bool
      • Optional
      • Defaults to undefined
      • If set to true, will force this column to be visible at all times.
      • If set to false will force this column to be hidden at all times.
    • width: Int
      • Optional
      • Defaults to 150
      • Specifies the width for the column
    • minWidth: Int
      • Optional
      • Defaults to 0
      • Specifies the minimum width for the column
    • maxWidth: Int
      • Optional
      • Defaults to 0
      • Specifies the maximum width for the column
    • cssClass: String
      • Optional
      • Adds css class to a header cell

    Row Options

    The following options are supported on any row object you can pass to data.

    • rowId: String
      • Required
      • Row identifier. It will be passed within event arguments to identify cell
    • cssClass: String
      • Optional
      • Adds css class for entire row
    • data: Object<[columnName]: Cell>
      • Optional
      • Describes cell values and their metadata's
      • Hash map where key is column name (must match to name property of the column) and value is cell options object

    Cell Options

    The following options are supported on any cell object you can pass to Row.

    • cssClass: String
      • Optional
      • Adds css class for entire row
    • type: <text|textedit|checkbox|radio>
      • Optional, if not provided then it's text by default
      • Defines how the cell must be rendered. It might be just a text or an editor

    • data: <{ value: any}>

      • Optional
      • Object with data which needs to be presented in a cell or cell editor
      • Value types (of data.value) by cell type:
        • text
          • string
        • textedit
          • string
        • checkbox
          • boolean
        • radio
          • boolean

    • editorOptions: <object>

      • isRequired <Boolean>
        • Optional
        • Draws required mark for the editor
      • isDisabled <Boolean>
        • Optional
          • Forces editor to be in disabled state
